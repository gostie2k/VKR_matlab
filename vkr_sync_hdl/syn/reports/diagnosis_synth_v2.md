# Этап 3.12 — итоговый протокол OOC-синтеза (v2)

**Дата:** 2026-05-20.
**Тулза:** Vivado 2023.2 (-mode batch).
**FPGA part:** xc7z020clg484-1 (Zynq-7020, CLG484, speed grade −1).
**Целевая частота:** 100 МГц (период 10.000 нс), контроль на 150 МГц.

## 1. Резюме

Три блокера, выявленные на первой итерации OOC-синтеза
(`syn/reports/blockers_3_12.md`), устранены точечными правками
RTL согласно постановке задачи 1 текущего этапа:

| Блокер             | Правка                                            | Эффект                          |
|--------------------|---------------------------------------------------|--------------------------------|
| sync_top.v:151     | `if (rst)` → `if (reset)`                         | синтез проходит                 |
| sync_mod1_nco.v    | `mu_full / w_step` → `mu_simple` (битовый сдвиг)  | WNS −47.27 → **+5.244 нс**     |
| sync_loop_filter_pi.v | pipeline-ступень (vi_reg ↔ выход)              | WNS −0.635 → **+2.915 нс**     |

V3-перепрогон (D6) функциональную эквивалентность подтвердил:
MER @ EbN0=20 = 13.77 dB (D5 = 13.88; деградация 0.11 dB),
MER @ EbN0=60 = 31.97 dB (D5 = 32.72), μ_lock @ 20 = 0.6141,
μ_lock @ 60 = 0.6099, квадранты 26.7/24.3/25.5/23.5, N_sym = 1980.
Все приёмочные критерии задачи 2.6 выполнены.

**Однако,** OOC-синтез на 100 МГц закрылся только в `sync_mod1_nco`
и `sync_loop_filter_pi`. В трёх других подмодулях
(`sync_ted_gardner`, `sync_agc`, `sync_farrow_parab`) и,
как следствие, в `sync_top` сохраняются timing-нарушения, не
относящиеся к трём блокерам задачи 1 (они присутствовали и до
правок и в `blockers_3_12.md` были отнесены к категории
«сопутствующих»). Согласно правилу 3.6 брифа, остановился без
дополнительных правок RTL. Список и критические пути — §5.

## 2. Сводка правок RTL

Полный diff трёх файлов сохранён в `syn/reports/rtl_fixes.diff`
(152 строки, изменения только в трёх файлах:
`rtl/sync_top.v`, `rtl/sync_mod1_nco.v`, `rtl/sync_loop_filter_pi.v`).

### 2.1. `rtl/sync_top.v` — одна строка

```diff
- if (rst) begin
+ if (reset) begin
```

Sensitivity list объявляет асинхронный фронт `reset`; в теле теперь
проверяется тот же сигнал. Семантика — delay-line строба
сбрасывается только аппаратным reset; soft-reset подмодули
получают через свои порты `.reset(rst)`, и delay-line
автоматически очищается за два такта.

### 2.2. `rtl/sync_mod1_nco.v` — замена блока деления

```diff
- wire [W_CNT + W_MU - 1:0] cnt_shifted = {cnt_reg, {W_MU{1'b0}}};
- wire [W_CNT + W_MU - 1:0] mu_full     = (w_step != 0) ? cnt_shifted / w_step : 0;
- wire [W_MU-1:0]           mu_calc     = mu_full[W_MU-1:0];
+ wire [W_MU-1:0] mu_calc = cnt_reg[W_CNT-2 -: W_MU];
+ // (закомментированный задел под Divider IP сохранён ниже)
```

Поведенческое деление через системный оператор «/» разворачивалось
в 136-уровневую комбинационную CARRY-цепочку, физически не
помещавшуюся в 10 нс. Аппроксимация `mu ≈ CNT_old · 2` через
битовый сдвиг точна при `w_step = W_nom = 0.5` (штатный режим
петли); погрешность зафиксирована в этапе 3.3 дневника.
Архитектурный задел сохранён в виде комментария.

### 2.3. `rtl/sync_loop_filter_pi.v` — pipeline-ступень

В ступени 2 длинная цепь `adder→sat→adder→sat→trunc → v_pi_out_reg`
разорвана на две ступени:

- **Ступень 2 (новая)**: только `vi_sum → vi_sat → vi_reg`
  (+ задержка `vp_s1 → vp_s1_d`, валид `valid_s2`).
- **Ступень 3 (новая)**: `v_pi_sum = vp_s1_d + vi_reg → sat → trunc → v_pi_out`
  по `valid_s2`.

Латентность PI выросла с 2 до 3 тактов; общая латентность
разомкнутой петли — с 8 до 9 тактов. Влияние на запас фазы
при `B_n·T = 0.02` и `ζ = 1/√2` — менее 4° (D6 подтвердил).
Округление `+ (1 << 14)` перед `>>>15` (этап 3.11), коэффициенты
K1/K2, формат Q1.15, двойное насыщение — без изменений.

## 5. Открытые блокеры (timing-фейлы вне задачи 1)

Эти три пути присутствовали в `blockers_3_12.md` как
«сопутствующие» (раздел «Сопутствующие timing-фейлы по подмодулям»)
и не входили в задачу 1 этого этапа.

### 5.1. `sync_farrow_parab` (−1.519 нс) и `sync_top` (−1.518 нс)

```
Источник:   t_mul_mu_i_i_18/C    (FDCE регистр на ступени 2 Горнера)
Назначение: xi_s3_i_reg[14]/D    (регистр выхода ступени 3)
Data path:  11.464 нс           (DSP48E1 product → big add → round/sat → reg)
Logic:      10 уровней (5 CARRY4 + 2 DSP48E1 + LUT2/4/6)
```

Содержательно (rtl/sync_farrow_parab.v:183–209):

```
W_MUL2 = 31
t_mul_mu_{i,q} = t_s2_{i,q} * mu_s2_sgn         ; W_MUL2 бит, DSP
v0_aligned_{i,q} = v0_s2_{i,q} <<< (2*W_MU)     ; sign-extend + shift
xi_scaled_{i,q} = t_mul_mu_{i,q} + v0_aligned_{i,q}   ; ШИРОКИЙ adder
round_const   = 1 << (SHIFT_OUT - 1)
xi_round      = xi_scaled + round_const         ; ещё один add
sat            = saturate(xi_round, ...)        ; CARRY-cmp
trunc          = xi_round >>> SHIFT_OUT
xi_s3_*_reg   <= trunc                         ; регистр
```

Между регистром t_s2 (выход DSP) и xi_s3-регистром стоит широкая
комбинационная цепочка add + add + sat — это и есть −1.5 нс.
Канонически разбивается одной pipeline-ступенью между
`xi_scaled` и `xi_round`/`sat` (по аналогии с правкой PI-фильтра
в задаче 1.3): +1 такт латентности Farrow с 3 до 4.

### 5.2. `sync_ted_gardner` (−0.224 нс)

```
Источник:   diff_s1_q_reg[16]/C  (регистр Δy = y_prev − y_curr)
Назначение: e_out_reg[3]/D       (регистр выходной ошибки)
Data path:  10.169 нс           (DSP product → CARRY add → reg)
Logic:      9 уровней (5 CARRY4 + 2 DSP48E1 + LUT1/5)
```

Цепь Gardner: e = y_mid_I·diff_I + y_mid_Q·diff_Q. Два DSP48E1
для умножений, затем CARRY-сумматор для агрегации I+Q. Маленький
WNS (0.22 нс) — обычно лечится включением выходного регистра DSP
(`(* USE_DSP = "yes" *)` уже стоит или `register_balancing`) либо
точечной pipeline-ступенью между двумя произведениями и их
суммой.

### 5.3. `sync_agc` (−0.433 нс)

```
Источник:   in_q[15]              (вход поток. данных — НЕ конфиг-порт)
Назначение: accum_reg[32]/D       (аккумулятор для AGC-power)
Data path:  9.384 нс              (square via DSP → CARRY add → 33-bit accum)
Logic:      14 уровней (11 CARRY4 + DSP48E1 + LUT2/3)
```

Цепь AGC: power = I² + Q², затем 33-битное накопление в
скользящем окне. Длинная CARRY-цепочка вызвана разрядностью
аккумулятора. Канонически — pipeline-вставка между квадратами
и накопителем.

### 5.4. Резюме по блокерам §5

| Модуль                | WNS    | Сложность правки                                  |
|-----------------------|--------|---------------------------------------------------|
| sync_ted_gardner      | −0.22  | retiming или +1 такт между DSP и финальным sum     |
| sync_agc              | −0.43  | pipeline между sq → accum                          |
| sync_farrow_parab     | −1.52  | pipeline в ступени 3 Горнера (между mul и add)     |
| sync_top              | −1.52  | автоматически устраняется правкой Farrow           |

`sync_top` критический путь = критический путь Farrow. То есть
правка только в Farrow (5.1) должна закрыть и `sync_top`.

## 6. Особенности OOC-ограничений

Конфигурационные порты от AXI-Lite shim в реальной интеграции
(Block Design + AXI-Lite wrapper) драйвятся статическими
регистрами в той же clock-области и обновляются один раз за
сеанс. В OOC-режиме при `set_input_delay 2.0 ns` они создавали
артефактный критический путь (этап 3.12, первая итерация). Для
устранения объявлены `set_false_path` в следующих XDC:

| XDC                                          | Порты с false_path                                                |
|----------------------------------------------|-------------------------------------------------------------------|
| `syn/constraints/sync_top_ooc.xdc`           | reset, ctrl_*, reg_k1/k2/w_nom/clamp/agc_target                   |
| `syn/constraints/sync_top_ooc_150mhz.xdc`    | то же                                                              |
| `syn/constraints/sync_loop_filter_pi_ooc.xdc`| reset, k1, k2, clamp_lim                                          |
| `syn/constraints/sync_agc_ooc.xdc`           | reset, agc_bypass, p_target                                       |

В integrated-режиме эти ограничения избыточны: регистры AXI-Lite
shim и регистры подмодулей принадлежат одной clock-области,
и timing между ними учитывается стандартными правилами
синхронного дизайна без дополнительных ограничений.

## 7. Особенности инференса

### 7.1. DSP48E1 — все умножения корректно инферированы

| Модуль                | Умножения        | DSP48E1 факт |
|-----------------------|------------------|--------------|
| sync_loop_filter_pi   | e·K1, e·K2       | 2            |
| sync_ted_gardner      | y_mid_I·diff_I, y_mid_Q·diff_Q | 2 |
| sync_farrow_parab     | μ·c1, μ·t, …     | 6            |
| sync_agc              | I², Q², gain·in_I, gain·in_Q | 4 |
| sync_mod1_nco         | (нет умножений)  | 0            |
| **sync_top (итого)**  |                  | **14**       |

Все произведения попали в DSP48E1, ни одно не вынесено в LUT.
В `sync_loop_filter_pi` округление `+ (1 << 14)` поглощено
пре-аддером DSP (DSP-output-register реализует `prod_k2_r`).

### 7.2. `sync_mod1_nco` — деление устранено

После замены `mu_full = cnt_shifted / w_step` на сдвиг
`mu_simple = cnt_reg[W_CNT-2 -: W_MU]`:
- 0 DSP48E1, 17 LUT (всего), 29 FF;
- критический путь: 7 уровней логики (5 CARRY4 для вычитающего
  счётчика + 2 LUT для underflow-детекции), WNS +5.244 нс,
  достижимая Fmax ≈ 210 МГц.

### 7.3. `sync_farrow_parab` — 3 такта pipeline сохранены

Иерархический отчёт показывает 331 FF в `u_farrow`, что
покрывает три pipeline-ступени Горнера (v_s1, t_s2, mu_s2,
vp_s1 и т. п. для I/Q). Retiming не разрушил архитектурную
конвейерность; задержка от `in_valid` до `out_valid` сохранена
(совпадает с задержкой `strobe_d2` в `sync_top`).

### 7.4. `sync_agc` — отсутствие свёртки

5 регистров под `accum_reg`, `gain_reg`, конвейерные регистры
квадратов — итого 66 FF. CORDIC-цепочки нет (AGC построен на
прямом ARM-методе через скользящее окно).

### 7.5. `sync_loop_filter_pi` — pipeline-ступень внедрена

После правки 2.3:
- 70 FF (было ~52): добавились `vi_reg` (32) + `vp_s1_d` (32)
  + `valid_s2` (1) + retiming-копии;
- 194 LUT (было 186): два сатуратора по 16 LUT + glue;
- 2 DSP48E1 — без изменений;
- DRC `XPM_CDC-1` (наличие CDC) — нет; модуль однотактный,
  пересинхронизация не требуется.

## 8. Критические warning'ы

`syn/reports/critical_warnings.txt` содержит:
- echo-строки `puts "ERROR: usage..."` из `run_ooc_synth.tcl` —
  ложные срабатывания парсера (это не реальный ERROR, а часть
  определения функции);
- `CRITICAL WARNING [Vivado 12-4739] set_false_path: No valid
  object(s) found` для подмодулей, синтезируемых с
  `sync_top_ooc.xdc` (Farrow, NCO, TED) — содержит false_path
  на порты `reg_k1`, `ctrl_*` и т. п., которых у этих подмодулей
  нет. Constraint становится no-op, синтез не страдает;
  для чистоты можно создать отдельные «без false_path» XDC,
  но это косметика, не влияющая на timing.

Реальных синтез-warning по RTL (`Synth 8-*` latches/sensitivity)
не обнаружено. Блокер №1 (Synth 8-91) устранён.

## 9. Открытые вопросы

1. **Закрытие timing в Farrow ступени 3 Горнера** — единственный
   путь, мешающий закрыть `sync_top` на 100 МГц. Канонически —
   pipeline-ступень между `xi_scaled` и `sat/trunc`, +1 такт
   латентности Farrow.

2. **Закрытие в `sync_ted_gardner`** — маленький WNS −0.22 нс,
   может закрыться retiming'ом без явной правки RTL
   (флаг `-retiming` в `synth_design`/`opt_design`). Альтернатива —
   pipeline между DSP и финальным sum.

3. **Закрытие в `sync_agc`** — WNS −0.43 нс, требует
   pipeline-ступени между квадратами и накопителем.

4. **Косметика XDC** — отдельные XDC для подмодулей без
   AXI-Lite-портов (Farrow, NCO, TED), чтобы убрать
   CRITICAL WARNING 12-4739.

5. **Контрольный прогон на 150 МГц** — пока упирается в Farrow,
   как и на 100 МГц. После правки Farrow ceiling возрастёт.

После согласования следующего шага (правка Farrow и/или AGC/TED)
можно повторить sweep и сформировать финальную сводку с
положительным WNS во всех модулях.
