# Этап 3.12, итерация 3 — D7-перепрогон провалился, нужно решение

**Дата:** 2026-05-20.
**Стимул:** τ = 0.3, EbN0 = 20 дБ, seed = 42, N_SYM = 2000 — штатный.

## TL;DR

Сделал три pipeline-вставки строго по постановке итерации 3
(`sync_farrow_parab.v`, `sync_ted_gardner.v`, `sync_agc.v`).
RTL правил ровно в трёх перечисленных файлах, `sync_top.v` не
трогал. V3-перепрогон (D7) дал **MER @ EbN0=20 = 12.51 дБ**,
что **ниже приёмочного порога 13.0 дБ**. По правилу 2.5
постановки — остановка.

Причина — увеличение латентности Farrow с 3 до 4 тактов
рассинхронизировало строб TED в `sync_top.v`. Нужно решение
о том, как продолжить.

## Корневая причина — `sync_top.v:144–157`

В `sync_top.v` зашита задержка `strobe_d2` ровно на 2 такта:

```verilog
// Задержка strobe на 2 такта (= латентность Фарроу от in_valid до out_valid)
// NCO и Farrow получают agc_valid одновременно. NCO выдаёт strobe через
// 1 такт (registered), Farrow выдаёт XI через 3 такта (3 pipeline stages).
// Разница: 3 - 1 = 2 такта → strobe_d2 совпадает с farrow_valid.
reg strobe_d1, strobe_d2;
always @(posedge clk or posedge reset) ... begin
    strobe_d1 <= nco_strobe;
    strobe_d2 <= strobe_d1;
end
```

Эта константа «2 такта» была откалибрована под старую латентность
Farrow = 3 такта (1 NCO + 2 delay = 3, совпадает с Farrow). После
правки итерации 3 Farrow стал 4 такта, а задержка строба
осталась 2 → `strobe_d2` приходит **на 1 такт раньше**, чем
соответствующий `farrow_valid`. TED захватывает y_curr / y_mid
со сдвигом 1 такта — Gardner работает на смещённых выборках:

- μ_lock = 0.6076 (близко к норме 0.61, петля заперта);
- vi_reg = +1.5 (стабильно, нет windup);
- e_out|mean = 3841 (ср. 3863 в D6, амплитуды похожи);
- но phase std вырос с 8.4° до 10.0° → дополнительный jitter
  ровно от того, что строб бьёт не в оптимальную точку.

Это классическая картина «petля заперта, но на subopt-точке выборки».

Пипка по AGC и TED таких эффектов не даёт:
- AGC +1 такт сдвигает downstream равномерно → выравнивание NCO ↔
  Farrow относительно `agc_valid` не нарушено.
- TED +1 такт идёт перед PI, у которого нет фиксированной
  задержки относительно него; +1 такт TED только увеличивает
  суммарную latency петли, что компенсируется loop bandwidth.

Ломает только Farrow +1, потому что между Farrow и NCO задержка
строба зашита **константой**, а не вычисляется.

## Варианты продолжения

### Вариант А — bookkeeping-правка `sync_top.v` (рекомендуется)

Добавить третий регистр задержки строба:

```verilog
// rtl/sync_top.v, заменить блок строк 149–157:
reg strobe_d1, strobe_d2, strobe_d3;
always @(posedge clk or posedge reset) begin
    if (reset) begin
        strobe_d1 <= 0; strobe_d2 <= 0; strobe_d3 <= 0;
    end else begin
        strobe_d1 <= nco_strobe;
        strobe_d2 <= strobe_d1;
        strobe_d3 <= strobe_d2;
    end
end
```

И заменить две ссылки `strobe_d2` → `strobe_d3` (одна на входе TED,
одна в `sym_valid`). Плюс актуализировать комментарий «4 − 1 = 3 такта».

**Плюсы:**
- Минимальная правка (1 регистр + 2 переименования).
- Архитектура алгоритма не меняется.
- Закрывает реальную проблему — рассинхронизацию.

**Минусы:**
- Формально нарушает «не трогать sync_top.v» из постановки,
  но содержательно — это техническое согласование задержки
  под новую латентность Farrow, не правка алгоритма.

**Ожидаемый эффект:** MER @ 20 восстанавливается до ≈ 13.7 дБ
(уровень D6 + малая поправка на +1 такт общей петли);
суммарная латентность петли = AGC 2 + Farrow 4 + TED 3 + PI 3 = 12 тактов
(как и заявлено в постановке).

### Вариант Б — переработка Farrow без увеличения латентности

Объединить новую pipeline-ступень с одной из существующих. Например:
- зарегистрировать `t_mul_mu_*` через DSP P-register, одновременно
  убрав регистр `t_s2_*` (т. к. DSP уже даёт регистрированный выход);
- сохранить общую латентность Farrow = 3 такта.

**Плюсы:**
- `sync_top.v` остаётся нетронутым.
- Латентность петли не растёт.

**Минусы:**
- Требует более глубокой переработки Farrow с риском регрессий.
- Без +1 такта Farrow закроется по timing менее надёжно
  (зависит от того, как именно DSP P-reg сможет поглотить
  следующий add).

### Вариант В — оставить только две из трёх правок

Например, не делать pipeline в Farrow, оставить только AGC и TED
+ pipeline в Farrow заменить на retiming-флаг `synth_design`:

```tcl
synth_design -top sync_farrow_parab -retiming on -mode out_of_context
```

`sync_top.v` остаётся нетронутым, латентность Farrow формально
не меняется. Но retiming в Vivado не всегда даёт предсказуемый
результат, особенно через границы DSP.
